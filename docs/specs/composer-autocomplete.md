# Spec: Composer Autocomplete

## 1. What

Autocomplete the Composer field on both the Add Piece and Edit Piece screens, drawing suggestions from the user's own piece history.

## 2. Why

Reduces typos and inconsistent composer spellings (e.g. "Chopin" vs "chopin") that would degrade any future filtering or grouping by composer. Speeds up data entry for users with many pieces by the same composer.

## 3. Data Model

No Firestore schema changes. The composer field on pieces remains a plain string. Suggestions are derived at runtime from the set of composer strings already saved across the user's pieces (loaded via existing `usePieces` hook).

## 4. UI Flow

1. User focuses the Composer TextInput.
2. As soon as they type 1+ character, the app filters the composer history (case-insensitive prefix/substring match) and shows up to 5 suggestions in a Paper `Menu` anchored below the field.
3. Tapping a suggestion fills the field with that suggestion's string and dismisses the menu.
4. Free-text always allowed — user can ignore suggestions and type anything.
5. Menu dismisses when: user taps outside, presses back, clears the field below 1 char, or selects a suggestion.

### Deduplication of suggestions

- Group composer strings case-insensitively.
- Within each group, display the most-recently-used variant (i.e. the composer string from the piece with the latest `updatedAt` / `createdAt`).
- When a suggestion is selected, store that exact suggestion string in Firestore (not a normalized form).
- Free-text entered without selecting a suggestion is stored exactly as typed.

### Alias handling

No disambiguation. If the user has both "Bach" and "J.S. Bach", both appear as separate suggestions. User decides which to use.

## 5. Component

Extract shared component: `components/ComposerAutocompleteInput.tsx`

```
Props:
  value: string
  onChangeText: (text: string) => void
  label: string
  error?: boolean
  helperText?: string
  pieces: Piece[]          // passed in from parent — already loaded
```

Used in both `app/(app)/piece/add.tsx` and `app/(app)/piece/[id]/edit.tsx`.

## 6. Logging

None. No practice signals generated by autocomplete interaction.

## 7. Out of Scope

- Curated composer list (use history only)
- Composer name normalization to canonical form
- Alias merging or disambiguation UI
- Fetching suggestions from any external API
- Autocomplete on any field other than Composer

## 8. Phases

### Phase 1: Component + Add screen

1. Create `components/ComposerAutocompleteInput.tsx` with Paper `Menu` + `TextInput`.
2. Derive sorted, deduplicated composer list from `pieces` prop (case-insensitive, most-recently-used variant).
3. Filter list by current input (case-insensitive substring match), cap at 5.
4. Wire into `app/(app)/piece/add.tsx` — replace bare `TextInput`/`HelperText` with the new component.
5. Run lint + tests.

### Phase 2: Edit screen

1. Wire `ComposerAutocompleteInput` into `app/(app)/piece/[id]/edit.tsx` — replace bare `TextInput`/`HelperText`.
2. Run lint + tests.

### Phase 3: Playwright end-to-end verification

1. Open `http://localhost:8082`.
2. Login with test credentials.
3. Add a piece with composer "Chopin" — verify suggestion appears after adding a second piece.
4. Edit a piece — verify existing composer pre-fills and suggestions appear.
5. Verify free-text (unlisted composer) saves correctly.
6. Verify menu dismisses on outside tap and on selection.
